## 0. TL;DR + the load-bearing honesty caveat

B7 is a two-part, mechanically-simple flip:
1. **7 recipe edits** — repoint each toolchain leaf's `replace_on_cycle` **Amd64** cycle-breaker
   `url`+`sha256` from the debian-derived prebuilt blob (`gs://minimal-staging-archives/prebuilts/...`)
   to our own **seed-rooted from-source artifact** in the release-channel bucket
   (`gs://minimalmertic-release-channel-stable/...`, already in `ALLOWED_HYDRATE_BUCKETS`).
2. **7 trust-config edits** (in BOTH copies) — relocate the `source/<leaf>-prebuilt@<ver>` entries
   from `bootstrap_deps` → `bootstrap_artifacts`, giving each an `applies_to_packages` + `url`, and
   swapping the sha from the prebuilt-blob sha to the from-source-artifact sha.

**The one thing that blocks turning this from "structurally exact" to "apply-ready":** the
from-source artifacts **do not exist yet.** Per `b4-b7-readiness.md` (lines 13–14, 35–39):
B4 (re-root of linux-headers/binutils/glibc/gmp/mpfr/mpc onto the R11 gcc-10.4.0 pivot) and B5
(rebuild production gcc-15.2.0 against from-source glibc) are **DESIGNED but NOT BUILT.** Until
B4/B5 seal and publish to the release channel, the **new `url` and `sha256` are unknowable.**
Every new value in the diffs below is therefore a **placeholder token** of the form

## 1b. DIFF — trust-config `bootstrap_deps` → `bootstrap_artifacts` relocation (BOTH copies)

`trust-config.json` and `hermetic-builder-rs/trust-config.json` are **byte-for-byte identical**
(`diff` = no output). This exact hunk pair applies to BOTH files; edit in lockstep or the builder
image (which vendors the `hermetic-builder-rs/` copy) diverges from the repo-root copy that Go paths
read.

**Schema note (`trust_config.rs:29-41`):** `bootstrap_deps` entries are `{name, sha256, rationale}`.
`BootstrapArtifact` requires `{name, applies_to_packages: [String], url, sha256}` (extra keys like
`rationale` are tolerated — `tolerates_unknown_artifact_fields` test, line 94). So each relocated
entry GAINS `applies_to_packages` + `url`, keeps `rationale`, and its `sha256` changes from the
prebuilt-blob sha to the **from-source-artifact sha** (same `<B4_*_SHA>`/`617e2057a63fc5d7840b752a8b1cdfdbef3ba193c36eed52ebb97396122cb148` tokens as §1a).

**`applies_to_packages` = the BARE from-source leaf name** (`"gcc"`, `"glibc"`, `"binutils"`,
`"gmp"`, `"mpfr"`, `"mpc"`, `"linux_headers"`) — NOT `"<leaf> (prebuilt)"`. Verified against the
recipes: the from-source spec `name` is bare (`gcc/build.ncl:32 name = "gcc"`, etc.); the breaker
spec `name` is `"<leaf> (prebuilt)"`. Pass-3 hydration (`breaker_hydrate.rs:209`) matches
`applies_to_packages` against `spec.name` and writes the artifact into that spec's `spec_hash` slot.
We want the **consumer-visible from-source slot** populated (§Caveat A), so the bare name is correct.
Using `"(prebuilt)"` would populate the breaker slot instead — redundant with Pass-1 and wrong.

## 2. CAVEAT A (twinning) — VERDICT: **CONFIRMED.** Pass-2 copies the prebuilt into an empty from-source slot.

**Code evidence — `hermetic-builder-rs/src/breaker_hydrate.rs`, Pass 2 (lines 156-186):**

- L168-184: otherwise `copy_dir_recursive(breaker_entry.path(), pending.path())` — **copies the

So the from-source leaf slot is filled from whatever `replace_on_cycle` points at, **labeled as the
from-source spec**, whenever that slot is empty at hydrate time. **Pre-B7 this silently ships debian
prebuilt bytes under the `gcc`/`glibc`/… from-source label** — exactly the failure the caveat names.

### How B7 forces the real (seed-rooted) build, not the prebuilt substitution
Two independent mechanisms, both pointing at the SAME release-channel from-source artifact:

1. **Repoint `replace_on_cycle` (§1a).** Pass 1 (L113-154) hydrates the breaker slot from its
   `Source` url — now `<B*_URL>` in `minimalmertic-release-channel-stable`, sha-gated at L133-135.
   Pass 2 then twins **from-source bytes** (not debian) into the from-source slot. The twin is now
   *correct by construction*: breaker bytes == from-source bytes.
2. **`bootstrap_artifacts` Pass-3 entry with `applies_to_packages=[bare leaf name]` (§1b).**
   Pass 3 (L198-285) resolves the bare name to the from-source spec (L209 `spec.name == pkg_name`),
   computes its `spec_hash` (L213), and if empty (L214) fetches+sha-verifies+writes the
   release-channel artifact into that exact slot. Note Pass 2 runs *before* Pass 3, so on a cold
   cache Pass 2's (now-correct) twin wins and Pass 3 no-ops (L214 read_dir ok → skip); the entry's
   durable job is the **signer subset check** (below), with hydration as belt-and-suspenders.

**Precise honesty about "forces a real build":** the hydration machinery cannot itself distinguish
"genuine B5 build output" from "prebuilt relabeled" — it copies bytes. The *guarantee* that the slot
holds seed-rooted bytes comes from **what is published at `<B*_URL>`**: a release-channel object
carrying a `.intoto.jsonl` whose `CHAIN_ENFORCE` predecessor-closure roots at the 229-byte hex0 seed.

any consumer cascade runs `hydrate_closure` (else the slot is empty and gets twinned — from the now
seed-rooted breaker, still fine — but you want the *authoritative* B5 build to be the published
bytes), and (b) `url` in all three places (leaf `replace_on_cycle`, breaker source, bootstrap_artifact)
resolves to that one attested object. **Residual danger = a stale/mis-pointed `url`** (e.g. left
pointing at the old prebuilt, or at a B4 rung that was itself twinned): verify with DUMP_CLOSURE_HASHES
(§3) that the leaf slot is populated by a `fetched=true, origin=release-channel` artifact.

## 3. CAVEAT B (spec_hash cascade) — VERDICT: **CONFIRMED — it cascades to every downstream consumer.**

Swapping a leaf's `replace_on_cycle` Amd64 arm **changes the leaf's `spec_hash`**, which changes the
`spec_hash` of every package whose build closure reaches that leaf. For gcc/glibc/binutils that is
effectively the **entire ~368-package catalogue**; gmp/mpfr/mpc/linux_headers reach the catalogue
transitively (gcc runtime_deps + glibc/linux-headers). This is the intended graph-wide
rebuild+re-attestation, **not** a no-op.

**Code evidence — `worktrees/minimal-fetcher-env-vars/crates/graph/src/spec_hasher.rs`:**
- **`process()` recurses INTO the breaker (L120-121):**
  `if let Some(replace_on_cycle) = build.replace_on_cycle.as_ref() {
   edges.push(Edge::ReplaceOnCycle(self.process(replace_on_cycle))); }`
  → `self.process(breaker)` allocates the breaker as its own entry in the `specs` array (L98-100).
- **`hash()` hashes EVERY spec in the array (L47-51):** `for (i,(bsr,edges)) in sh.specs...{ ...
  build_attrs_hash(build, &mut h); ... }` — the breaker is in `specs`, so it gets `build_attrs_hash`.
- **`build_attrs_hash` hashes the breaker's Source url+sha (L241-245 → L164-182):** it feeds each
  non-`Build` build_dep through `build_input_hash`, which for a `Source(SourceFetch::Web{url,sha256})`
  writes `b"url" + url.bytes + sha256.bytes` (L179-181). The breaker's only build_dep is its
  prebuilt `Source` → **its url+sha ARE in the hash.**
- **The `ReplaceOnCycle` edge also contributes (L80-83):** marker `b"c"` + the breaker's array index.

Therefore: change the Amd64 breaker `url`+`sha256` → the breaker spec's `build_attrs_hash` bytes
change → the leaf's overall `spec_hash` changes → every consumer that recurses through
`build_deps → leaf → replace_on_cycle → breaker` gets a new `spec_hash`. Cascade CONFIRMED.

**Why only the Amd64 arm matters:** the recipe's `match { {arch='Amd64,..}=>… {arch='Arm64,..}=>…}
target` resolves to exactly ONE `Source` at Nickel-eval time per target. In the amd64 graph the
breaker's `build_deps` contains only the amd64 Source; the arm64 arm is absent from the amd64 spec
entirely, so it never enters the amd64 `spec_hash`. Swapping only Amd64 is correct and complete for
the amd64 cutover (arm64 is milestone #12).

### Empirical confirmation test (DUMP_CLOSURE_HASHES) — do this on a THROWAWAY edit before flipping
`breaker_hydrate.rs:104-111`: when `DUMP_CLOSURE_HASHES=1` is set, `hydrate_closure` prints one line
per closure spec: `[closure] <spec.name> = <spec_hash hex>` (no fetches, cheap).

Procedure:
1. On a clean tree, evaluate the closure of a representative consumer (e.g. `coreutils`, whose
   closure reaches gcc/glibc/binutils) and a leaf itself (`gcc`), with `DUMP_CLOSURE_HASHES=1`, and
   save the output → `before.txt`.
2. Make a **throwaway** edit to ONE leaf's Amd64 `replace_on_cycle` url+sha (e.g. bump gcc's sha by
   one hex nibble — do NOT commit).
3. Re-run the same closure dump → `after.txt`.
4. `diff before.txt after.txt`. **Expected: the `gcc` line AND every downstream consumer line
   change**; lines for unrelated leaves you did not touch (until you swap them too) stay stable.
   A NON-empty diff spanning the catalogue = cascade confirmed = the intended graph-wide
   re-attestation. An empty/one-line diff would REFUTE the cascade (it does not — code says it will).

This is the go/no-go gate: it proves, empirically, that B7 triggers the full-catalogue rebuild you

### Signer-side consequence of the relocation (why bootstrap_artifacts, not bootstrap_deps)
`collect_declared_sources` + the subset invariant (breaker_hydrate.rs:328-332): the signer enforces
`resolvedDependencies ⊆ declaredSources ∪ bootstrap_deps ∪ bootstrap_artifacts ∪ release-channel`.

release-channel arm — but relocating the 7 into `bootstrap_artifacts` makes them **explicit,

than the soft `{name,sha}` bootstrap_deps list. Leaving the old prebuilt-sha bootstrap_deps entries
in place would be actively wrong: they pin bytes (the debian blobs) B7 no longer fetches — stale
trust surface. Relocation both removes the stale pin and installs the from-source anchor.

## 5. OBSERVATIONS (beyond task scope, flagged per instructions)

- **B7 is not yet actionable — it is blocked on B4+B5, which are DESIGNED but NOT BUILT**
  (`b4-b7-readiness.md:13-14,35-39`). This doc is the flip recipe; the load-bearing url+sha values

  patch. Anyone reading the task as "just relocate some JSON" is under-scoping it.
- **The task framing "re-root the 7 prebuilts" conflates B4/B5/B7.** Per the readiness doc the 7 are
  split: B4 re-roots 6, B5 handles gcc-15.2.0 (the 7th, self-hosting fixed point, NOT byte-identical
  to the old prebuilt by design), B7 is the cutover. Any repro/DDC gate written to expect byte-parity
  with the old prebuilt will false-fail on gcc (risk #3). The diffs here correctly treat gcc's new
  sha as a fresh from-source value, not the old `16a7b18a…`.
- **Pass-2-before-Pass-3 ordering is safe ONLY because we repoint replace_on_cycle too.** If a future
  change relocated to bootstrap_artifacts *without* also repointing the breaker, Pass 2 would twin the
  stale breaker (debian) bytes before Pass 3 could install the from-source artifact — silent
  regression. The two edits (§1a recipe + §1b trust-config) are a matched pair; never ship one alone.
  Consider adding a builder-side assertion that a leaf's `replace_on_cycle` url and its matching
  `bootstrap_artifacts` url are equal — cheap invariant that would have caught this class.
- **`applies_to_packages` name-matching is a stringly-typed footgun.** The from-source (`"gcc"`) vs
  breaker (`"gcc (prebuilt)"`) distinction is load-bearing and invisible to a reviewer skimming the
  JSON. A typo silently populates the wrong slot (or no slot — Pass-3 logs "no matching spec" and
  moves on, L266-277, non-fatal). Worth a lint that every `applies_to_packages` name resolves to a
  real spec in the target closure at build time.
- **Two byte-identical trust-config copies is itself a footgun** (one is vendored into the image, one
  is read by Go). They can drift silently. A single source-of-truth + a build-time `diff` assertion
  (or a symlink) would remove a whole class of "edited one, forgot the other" bugs.
